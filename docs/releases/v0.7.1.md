# v0.7.1 — Firebase hardening and communication templates

## Summary

`v0.7.1` is the stabilization and platform-hardening follow-up to the Firebase-native `v0.7.0` release. This release reduces duplication in profile access, versions the Firebase security/config layer, and prepares the communication/template surface needed for production-grade account emails and user messaging.

## Completed so far

- Added versioned Firebase project config files:
  - `firebase.json`
  - `firestore.rules`
  - `firestore.indexes.json`
  - `storage.rules`
- Added `src/hooks/useCurrentUserProfile.js` to centralize:
  - Firebase profile reads
  - Firestore + Firebase Auth profile saves
  - avatar validation and upload handling
- Refactored `ProfilePage` and `AccountCustomizePage` to reuse the same current-user profile hook.
- Updated top-level Firebase setup documentation in `README.md`.

## Included in this release

### Firebase communication templates

This release now explicitly includes a template pack for Firebase-driven user communications.

Scope includes:

- Firebase Auth email template source files for:
  - email verification
  - password reset
  - email recovery
- App-managed email template source files for:
  - welcome / confirmed account
  - new message alert
  - support acknowledgement
- In-app notification template definitions for common Life. events.
- Documentation for how to copy native Auth templates into Firebase and where app-managed templates will plug in later.
- Standard support contact for these templates:
  - `life.customer.support@gmail.com`

### Hardening fixes included

- Removed private contact fields from the publicly readable `profiles` document shape.
- Kept public contact visibility controlled through `public_email`, `public_phone`, `email_public`, and `phone_public` fields.
- Fixed Inventors & Investors media uploads so runtime storage paths match the new Firebase Storage rules.

## Notes

Firebase Authentication supports built-in template customization for core auth flows, but a post-confirmation "welcome" message and other app lifecycle emails will require an app-managed sender path (for example a Cloud Function or email extension/provider). This release includes the source templates and wiring plan so that work is tracked instead of living as a future sticky note.
