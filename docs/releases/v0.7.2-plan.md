# v0.7.2 plan — AI-enabled workflow, messaging delivery, and product reliability

## Goal

Turn the current `v0.7.1` Firebase baseline into a more reliable, more automated, and more polished release by focusing on three themes at once:

1. stronger delivery and messaging infrastructure
2. safer and cleaner networking/profile data boundaries
3. better developer productivity through editor and AI workflow integration

## Release themes

### 1) Communication delivery actually goes live

#### Communication objective

Move from “templates exist” to “the app can send and track the right messages.”

#### Communication work items

- Choose and wire an app-managed email delivery path for non-Firebase-auth messages.
- Send post-confirmation welcome emails after successful account verification.
- Send support acknowledgement emails when users contact support.
- Add optional new-message notifications for Investors & Inventors conversations.
- Centralize template rendering inputs so placeholders cannot silently drift.
- Introduce a single support contact source instead of repeating hardcoded values.

#### Communication acceptance signals

- Verification email remains Firebase-native.
- Welcome/support/message templates can be rendered from one shared shape.
- Missing placeholder data fails visibly in development.
- Support contact appears from configuration, not hardcoded in multiple files.

### 2) Networking and profile data become easier to trust

#### Networking objective

Finish the public/private profile separation so discovery data stays safe and predictable.

#### Networking work items

- Audit all `profiles`, `investorProfiles`, and `inventorProfiles` reads/writes.
- Ensure public discovery never depends on private-only fields.
- Replace remaining ad hoc client timestamps with a deliberate timestamp strategy.
- Add migration/cleanup logic for any stale `email` / `phone` fields still left in Firestore.
- Review conversation and unread-count refresh logic for redundant reads.
- Add safer fallback handling for partially saved networking profiles.

#### Networking acceptance signals

- Public discovery cards render without private fields.
- Existing users with older profile docs still load cleanly.
- Message/unread state remains stable across refreshes.

### 3) In-app notifications stop being mostly static

#### Notification objective

Start using the source-controlled notification catalog in real app flows.

#### Notification work items

- Replace hardcoded notification copy with template-driven entries where practical.
- Map key events to notification templates:
  - welcome confirmed
  - profile updated
  - support acknowledged
  - new message
  - streak celebration
- Add lightweight notification creation helpers instead of page-local duplication.
- Preserve local/offline resilience.

#### Notification acceptance signals

- Notification text comes from one shared catalog for supported events.
- New notification types can be added without editing multiple UI surfaces.

### 4) VS Code + AI workflow becomes first-class for the repo

#### Workspace objective

Make the workspace itself AI-friendly so future implementation passes are faster and more consistent.

#### Workspace work items

- Keep a shared `.vscode/mcp.json` for trusted workspace MCP servers.
- Recommend Copilot Chat and MCP-capable extensions in `.vscode/extensions.json`.
- Keep MCP-related editor settings versioned in `.vscode/settings.json`.
- Add a short repo note for how teammates should trust/start MCP servers in VS Code.
- Evaluate whether one extra repo doc is needed for “AI working agreement” and tool usage.

#### Workspace acceptance signals

- Opening the repo in VS Code surfaces the intended AI toolchain automatically.
- Workspace MCP servers are discoverable without manual JSON hunting.

### 5) Release hardening and confidence

#### Release objective

Reduce regression risk so future releases are less manual and less surprising.

#### Release work items

- Add at least a lightweight smoke-test plan for auth, profile, networking, and notifications.
- Review lint/build scripts for any missing validation step worth adding later.
- Expand release notes and handoff material with known follow-ups and deployment steps.
- Confirm Firebase rules/index deployment steps are documented for maintainers.

#### Release acceptance signals

- `v0.7.2` has a short repeatable validation checklist.
- Repo docs clearly describe what must be deployed versus what is only source-controlled.

## Suggested delivery order

1. ship AI/editor workflow support
2. wire app-managed email delivery
3. integrate template-driven notifications
4. finish profile/networking cleanup
5. perform release hardening and final QA

## Risk watchlist

- Email delivery adds provider/config complexity.
- Public/private profile cleanup can expose old data-shape assumptions.
- Notification integration can create duplicate events if done in multiple places.
- MCP setup should stay useful without forcing every contributor into the same AI stack.

## Release definition of done

- App-managed email flow exists for at least welcome + support acknowledgement.
- Notification templates are used in at least the highest-value live flows.
- Networking profile boundaries remain safe and predictable.
- VS Code workspace opens with a better AI integration path out of the box.
- Lint/build pass and release docs reflect the finished scope.
